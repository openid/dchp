# How to contribute

Thank you for your interest in the work of the Digital Credentials Harmonized
Presentation (DCHP) Working Group. Contributions from implementers and other
interested parties are what make the specifications better.

Here are some important resources:

- The [DCHP Working Group Home Page](https://openid.net/wg/digital-credentials-harmonized-presentation-working-group/)
  gives you background on the group, its charter, and its meeting schedule
- The [Working Procedures](https://openid.net/wp-content/uploads/2026/06/DCHP-WG-Exhibit.pdf)
  define how the group operates — roles, meetings, decision-making, and the
  specification development process. The key points for contributors are
  summarized below, but the Working Procedures are authoritative
- Join the [mailing list](https://lists.openid.net/mailman/listinfo/openid-specs-dchp)
  to follow the group's announcements and discussions
- Found a problem, or have a proposal? The
  [issue tracker](https://github.com/openid/dchp/issues) is the place to raise it
- The co-chairs can be contacted at
  [openid-specs-dchp-owner@lists.openid.net](mailto:openid-specs-dchp-owner@lists.openid.net)

## Intellectual Property Rights (IPR)

You must sign the OpenID Foundation's IPR Contribution Agreement before
contributing — this includes filing issues, opening pull requests, and
posting to the mailing list, not just contributing text. When completing the agreement, select either
"All WGs" or "DCHP WG" as the working group.

[More info on IPR and the contribution agreement](https://openid.net/intellectual-property/)

## Code of Conduct

All participants in the working group's activities — meetings, the mailing
list, and this repository — are expected to follow the OpenID Foundation's
[Code of Conduct](CODE_OF_CONDUCT.md).

## How the group works

GitHub is the working group's primary tool for developing the technical
specifications, and this repository is public. Decisions are made by group
consensus, normally in working group meetings, and important decisions
(including the decision to publish) are tracked in GitHub issues.

## Filing issues

- Open an issue to raise a topic for discussion by the group, and comment on
  existing issues to facilitate the discussion
- Search the existing issues first to avoid duplicates
- Keep each issue focused on a single topic — file separate issues for
  separate concerns
- Where possible, propose concrete text; concrete proposals are easier to
  discuss and adopt than open-ended concerns
- Issues the working group decides to close are marked "pending close" and
  are only formally closed after 14 days without objections

## Pull requests

- Pull requests for substantive changes should be opened once consensus on
  the normative changes has been reached in the corresponding issue; if
  significant discussion happens on a pull request, it may be closed and
  split into issues
- Editorial fixes (typos, broken links, formatting) can be submitted
  directly; the GitHub interface supports creating pull requests using the
  Edit (✏) button
- Reference the issue(s) a pull request resolves in its description
- Before a pull request is merged, the Working Procedures require that:
  - the pull request has been open for at least 2 weeks (not required for
    editorial pull requests)
  - there have been no normative changes to it in the last 2 weeks
  - the issue or pull request has been discussed in at least one working
    group meeting
  - there are no unresolved comments
  - it has 3 approvals
